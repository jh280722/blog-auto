1. 확장 프로그램 연결

권장 순서: PREPARE_EDITOR → WRITE_POST
`PREPARE_EDITOR`는 이제 content script alive만 보지 않고 실제 editor body/TinyMCE/contenteditable까지 확인합니다. fresh 진입은 기본적으로 `manage → newpost` 경로를 탑니다. `editor_not_ready`가 오면 `diagnostics.attempts[].editorProbe`, `diagnostics.entryPath`, `contentScriptAlive`, 그리고 `WRITE_POST` 실패 시 `preflight.reason`을 먼저 확인하세요.
`captcha_required`가 오면 먼저 응답의 `captchaArtifacts`, `solveHints`, 그리고 `stage` / `phase` / `publishTrace`를 확인하고, 필요하면 DIRECT STATE / CAPTCHA ARTIFACTS로 막힌 탭과 캡차 이미지를 다시 가져오세요. 응답의 `captchaContext.preferredSolveMode`와 `GET_DIRECT_PUBLISH_STATE(includeCaptchaContext: true)`의 값은 서비스워커가 frame scan까지 merge한 최종 힌트입니다. solve 뒤 iframe 껍데기만 남아 있으면 wait/resume은 이 merged context 기준으로 unblock 됩니다. v1.8.14부터는 `GET_DIRECT_PUBLISH_STATE`가 `directPublishRuntimeState`도 함께 내려주므로, `RESUME_DIRECT_PUBLISH({ waitForCaptcha: true })`가 worker restart 뒤 same-tab CAPTCHA wait를 자동으로 이어갈 예정인지 `active / nextCheckTimeMs / deadlineMs`로 바로 볼 수 있습니다. 또한 기본 handoff와 `GET_CAPTCHA_ARTIFACTS`는 이제 `artifacts.sourceImage`를 함께 채우므로, `frameDirectImage`가 있어도 원본 CAPTCHA 이미지를 바로 OCR/비전에 넘길 수 있습니다. 가벼운 응답이 필요할 때만 includeSourceImage: false로 opt-out 하세요. 이제는 `solveHints.prompt`를 그대로 OCR/비전 프롬프트로 쓰는 것이 기본입니다. `solveHints.submitField === "ocrTexts"`면 전체 단어 후보를 읽어 `INFER_CAPTCHA_ANSWER` 또는 `SUBMIT_CAPTCHA_AND_RESUME({ ocrTexts })`로 넘기고, `solveHints.submitField === "answer"`면 instruction/map 문제에서 읽은 전체 명칭을 바로 `SUBMIT_CAPTCHA_AND_RESUME({ answer })`에 넣으세요. v1.8.10 기준으로는 instruction/map 문제도 OCR 후보가 여러 줄이면 `SUBMIT_CAPTCHA_AND_RESUME({ ocrTexts })`에서 target entity 기준 자동 선택을 먼저 시도하고, 같은 challenge가 유지되는 동안에는 instruction/map 후보도 상위 답안을 순차 재시도합니다. 이때 재시도 직전에는 answer input / submit button이 다시 actionable 해질 때까지 짧게 재대기해 첫 오답 직후 transient captcha_input_not_found로 다음 후보를 놓치는 회차를 줄입니다. 또한 `challengeText / challengeSlotCount`뿐 아니라 `answerCandidates`도 같이 내려와서 `백촌오피스□ + 백촌오피스텔 → 텔` 같은 패턴을 자동으로 줄입니다. challenge 문구를 다시 못 읽는 경우에도 visual signature fallback으로 같은 challenge 여부를 계속 비교합니다. v1.8.11부터는 visible submit button이 잠깐 사라져도 submitApiAvailable가 true면 window.dkaptcha.submit() same-tab 경로를 계속 사용합니다. iframe-only / cross-origin CAPTCHA도 확장이 same-tab frame solve를 먼저 시도합니다. 정말 captcha_browser_handoff_required가 남을 때만 browser/CDP fallback + `RESUME_DIRECT_PUBLISH({ waitForCaptcha: true })`를 사용하세요. 이 wait 모드는 저장된 blocked tab이 아직 captcha_present여도 같은 탭을 그대로 잡고 대기한 뒤, CAPTCHA 해제 후 same-tab probe를 거쳐 재개합니다. v1.8.14부터는 이 browser handoff wait도 storage + `chrome.alarms`로 남아서 MV3 service worker restart 뒤 자동 복구됩니다. handoff 중 일시적인 editor_not_ready / frame scan miss는 clear로 치지 않고 saved tab polling을 계속합니다. v1.8.10부터는 post-CAPTCHA publish layer가 열린 탭에서도 draft snapshot이 MAIN world editor summary를 함께 합산해 false draft_restore_failed를 더 줄입니다.
이제 큐 항목도 `SUBMIT_CAPTCHA_AND_RESUME`가 same-tab 재개까지 직접 이어집니다. `GET_QUEUE`로 `captcha_paused` 항목의 id / captchaTabId뿐 아니라 captchaContext / solveHints / lastCaptchaArtifactCapture / lastCaptchaSubmitResult / lastCheckedAt까지 함께 확인한 뒤, `SUBMIT_CAPTCHA_AND_RESUME({ id, ocrTexts })` 또는 `SUBMIT_CAPTCHA_AND_RESUME({ tabId, answer })`를 호출하면 답안 제출 뒤 해당 큐 항목만 재개합니다. v1.8.21부터는 이 queue submit path가 저장된 captchaContext / solveHints를 answer inference에 우선 재사용하므로, worker restart 직후나 live context probe가 잠깐 비는 회차도 challengeText 기반 OCR 축약이 더 잘 이어집니다. direct publish state가 없고 `captcha_paused` 항목이 하나뿐이면 selector를 생략해도 자동 선택하지만, 여러 개면 id 또는 tabId를 명시하세요. v1.8.18부터는 queue resume도 publish_layer_open을 무조건 stale 탭으로 보지 않고, after_final_confirm 회차는 post-submit settle/navigation을 먼저 확인한 뒤 same-tab 재개를 이어갑니다. 구버전 호환이 필요할 때만 `SUBMIT_CAPTCHA` → `RESUME_DIRECT_PUBLISH`를 따로 사용하세요.

JavaScript에서 아래 코드로 확장 프로그램에 데이터를 보낼 수 있습니다.

const EXTENSION_ID = "YOUR_EXTENSION_ID"; // 1) 에디터 준비 (실제 editor body 준비까지 확인) chrome.runtime.sendMessage(EXTENSION_ID, { action: "PREPARE_EDITOR", data: { blogName: "your-blog-name" } }, (response) => console.log(response)); // 2) 단건 발행 chrome.runtime.sendMessage(EXTENSION_ID, { action: "WRITE_POST", data: { blogName: "your-blog-name", title: "글 제목", content: "<p>본문 HTML</p>", category: "카테고리명", tags: ["태그1", "태그2"], visibility: "public", images: [{url: "https://..."}], autoPublish: true } }, (response) => { console.log(response); // captcha_required면 blocked tab/state 확인 if (response.status === "captcha_required") { chrome.runtime.sendMessage(EXTENSION_ID, { action: "GET_DIRECT_PUBLISH_STATE", data: { includeCaptchaContext: true } }, (state) => console.log(state.directPublish?.captchaContext?.preferredSolveMode, state.solveHints, state)); } // editor_not_ready면 diagnostics/preflight로 실패 원인 확인 if (response.status === "editor_not_ready") { console.log(response.preflight || response.diagnostics); } }); // CAPTCHA 컨텍스트 직접 조회 chrome.runtime.sendMessage(EXTENSION_ID, { action: "GET_CAPTCHA_CONTEXT" }, (response) => console.log(response.solveHints, response)); // CAPTCHA 이미지 아티팩트 조회 (saved blocked tab 우선) chrome.runtime.sendMessage(EXTENSION_ID, { action: "GET_CAPTCHA_ARTIFACTS" }, (response) => console.log(response.solveHints, response.artifact)); // CAPTCHA 답안 제출 + 같은 탭 재개 (권장) // ocrTexts를 주면 answerCandidates 상위 순서대로 같은 challenge에서 자동 재시도 chrome.runtime.sendMessage(EXTENSION_ID, { action: "SUBMIT_CAPTCHA_AND_RESUME", data: { ocrTexts: ["백촌오피스텔", "백촌오피스탤"] } }, (response) => { if (response.status === "captcha_browser_handoff_required") { // rare fallback only chrome.runtime.sendMessage(EXTENSION_ID, { action: "RESUME_DIRECT_PUBLISH", // saved blocked tab 유지 data: { waitForCaptcha: true, waitTimeoutMs: 120000, pollIntervalMs: 1000 } }, (resume) => console.log('resume after browser handoff', resume.captchaWait, resume)); return; } console.log(response.submitResult, response.resumeResult || response); }); // 구버전 호환: CAPTCHA 답안만 제출 chrome.runtime.sendMessage(EXTENSION_ID, { action: "SUBMIT_CAPTCHA", data: { answer: "1234" } }, (response) => { console.log(response); if (!response.captchaStillAppears) { chrome.runtime.sendMessage(EXTENSION_ID, { action: "RESUME_DIRECT_PUBLISH", // saved blocked tab 유지 data: { waitForCaptcha: true, waitTimeoutMs: 120000, pollIntervalMs: 1000 } }, (resume) => console.log(resume)); } }); // 큐에 추가 chrome.runtime.sendMessage(EXTENSION_ID, { action: "ADD_TO_QUEUE", data: [{ title: "...", ... }] }, (response) => console.log(response)); // 큐 처리 시작 chrome.runtime.sendMessage(EXTENSION_ID, { action: "START_QUEUE" }, (response) => console.log(response));
